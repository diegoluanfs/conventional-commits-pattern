Padrão Conventional Commits

Introdução

O padrão Conventional Commits é uma especificação que define uma maneira simples de estruturar mensagens de commit, facilitando a leitura e compreensão tanto por humanos quanto por máquinas. Essa padronização melhora a comunicação entre desenvolvedores e mantém o histórico do repositório organizado.

Estrutura de uma mensagem de commit

                <tipo>(<escopo opcional>): <assunto>
                <corpo opcional>
                <rodapé opcional>
  • tipo: Define a categoria da mudança, como feat para nova funcionalidade ou fix para correção de bug.
  • escopo: Opcional; especifica a parte do código afetada.
  • assunto: Descrição breve da mudança, escrita no modo imperativo.
  • corpo: Opcional; fornece detalhes adicionais sobre a mudança.
  • rodapé: Opcional; inclui informações como referências a issues ou notas de breaking changes.

Tipos comuns de commits

  • feat: Adiciona nova funcionalidade ao código.
    Exemplo: feat: adicionar endpoint de autenticação
  • fix: Corrige um bug no código.
    Exemplo: fix: corrigir erro no cálculo de taxas
  • docs: Alterações na documentação.
    Exemplo: docs: atualizar README com instruções de instalação
  • style: Mudanças que não afetam o significado do código (espaços, formatação, etc.).
    Exemplo: style: ajustar indentação no arquivo de configuração
  • refactor: Alteração no código que não corrige bug nem adiciona funcionalidade.
    Exemplo: refactor: modularizar função de validação
  • test: Adiciona ou corrige testes.
    Exemplo: test: adicionar testes para o serviço de pagamento
  • chore: Atualizações de tarefas de build ou ferramentas auxiliares.
    Exemplo: chore: atualizar dependências do projeto
  • perf: Melhorias de performance no código.
    Exemplo: perf: otimizar consulta ao banco de dados
  • BREAKING CHANGE: Indica mudanças que quebram compatibilidade.
    Exemplo: feat!: remover suporte ao formato XML

Exemplos específicos

  • Alteração em um endpoint existente:
    Exemplo: feat(api): alterar o endpoint de login para suportar autenticação OAuth
  • Correção de bug em endpoint:
    Exemplo: fix(api): corrigir resposta incorreta do endpoint de consulta de pedidos

Uso do Commitlint

O Commitlint é uma ferramenta que valida se as mensagens de commit seguem o padrão Conventional Commits. Ele é integrado ao processo de desenvolvimento, garantindo que todos os commits estejam em conformidade.

Instalação e configuração

  1. Instale o Commitlint e o Husky:
    npm install --save-dev @commitlint/{config-conventional,cli} husky
  2. Crie o arquivo de configuração commitlint.config.js:
    module.exports = { extends: ['@commitlint/config-conventional'] };
  3. Adicione um hook de pré-commit com o Husky:
    npx husky add .husky/commit-msg 'npx --no-install commitlint --edit "$1"'

Benefícios do uso do Conventional Commits

  • Padronização: Mensagens consistentes facilitam a compreensão do histórico do projeto.
  • Automação: Permite a geração automática de changelogs e facilita o versionamento semântico.
  • Clareza: Ajuda a identificar rapidamente o propósito de cada commit.
